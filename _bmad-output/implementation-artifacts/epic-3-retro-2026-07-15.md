# Epic 3 Retrospective: Verify Folder Integrity

## Part 1: Epic Review

### Epic Goal

Enable users to verify a registered folder and receive clear, deterministic classifications for unchanged, modified, missing, new, moved/renamed, duplicate, and unreadable files.

### Completed Stories

- Story 3.1 - Compare Basic File States (`done`)
- Story 3.2 - Detect Moved or Renamed Files (`done`)
- Story 3.3 - Include Duplicate and Unreadable Findings in Verification (`done`)
- Story 3.4 - Wire `verify <folder>` Command and Reporting (`done`)

Epic 3 is `done`: 4 of 4 stories completed and reviewed.

### Major Capabilities Delivered

- Pure comparison of trusted baselines with current snapshots using SHA-256 as the content authority.
- Typed findings for unchanged, modified, missing, new, moved/renamed, and ambiguous moved/renamed files.
- Deterministic duplicate groups and separate unreadable findings without fabricated hashes.
- End-to-end `folderprint verify <folder>` orchestration, deterministic reporting, and V1 exit-code mapping.
- Safe `lastVerifiedAtUtc` persistence for reliable clean and differences-found results without changing the trusted baseline.
- Catalog lookup and validation that reject malformed or ambiguous persisted state.

### Important Design Decisions

- Keep `VerificationService` pure and keep command sequencing, writers, and exit codes in the CLI boundary.
- Compare paths and hashes ordinally and materialize independently owned, deterministically ordered result collections.
- Reconcile only unique cross-side hash matches as moved/renamed; retain every affected path and add a pathless marker when matching is ambiguous.
- Keep duplicate groups and unreadable paths separate from `FileChange` entries so distinct integrity signals are not conflated.
- Use `VerificationResult.HasDifferences` rather than formatted text to select the process exit code.
- Persist the verification timestamp before printing success, and reject the save if the catalog changed after the initial load.

### Review Findings That Improved the Implementation

- Story 3.2 added missing cross-subfolder, same-path precedence, one-sided repeat, `HasDifferences`, and result-shape coverage.
- Story 3.3 qualified duplicate groups by fingerprint count before path deduplication, added a full ordinal path-sequence tie-breaker, and tested nested group determinism directly.
- Story 3.4 added optimistic catalog conflict detection; rejected JSON `null`, malformed fingerprints, unsafe paths, and duplicate persisted paths; distinguished root disappearance from child traversal failures; and completed report ordering with a message tie-breaker.
- Review also kept story metadata and sprint tracking synchronized, making workflow state reliable.

### Lessons Learned

- Determinism must be specified through the final tie-breaker and tested with shuffled, nested, and repeated inputs.
- Ambiguity should remain lossless and typed; convenient pairing is incorrect when evidence is insufficient.
- Validate persisted data at the trust boundary before comparison or mutation.
- Catalog updates need conflict-aware persistence, even for metadata-only changes.
- Result-shape tests catch defects that flattened output assertions miss.
- Gated story sequencing worked: each story extended a reviewed contract without pulling later command or mutation scope forward.

### Risks Carried Forward

- All future catalog mutations must preserve conflict detection and must never overwrite malformed or concurrently changed state.
- Cross-platform catalog-path and relative-separator policy remains deferred and must not be changed incidentally.
- Story 4.3 refresh can accidentally blur verification and trust-baseline replacement; identity and failure atomicity require explicit tests.
- Story 5.1 `DuplicateFinder` must remain semantically consistent with verification duplicate grouping without coupling the commands.
- Unreadable findings currently carry paths only, so future reason-bearing output would require an intentional model decision.
