# Epic 5 Retrospective: Find Duplicate Files On Demand

## Part 1: Epic Review

### Epic Goal

Allow users to find duplicate readable files in any existing folder by SHA-256 content hash, without registration or catalog dependency, while preserving deterministic output and the Core/CLI boundary.

### Completed Stories

- Story 5.1 - Implement `DuplicateFinder` for Current Snapshots (`done`, reviewed)
- Story 5.2 - Wire `duplicates <folder>` Command (`done`, reviewed)
- Epic 5 is `done`: 2 of 2 stories completed and reviewed.
- All FolderPrint V1 functional stories are now complete.
- Final review validation passed 33 focused tests and all 243 Release tests, with a clean Release build, formatting, dependency, boundary, scope, safety, and diff checks.

### Major Capabilities Delivered

- Story 5.1 established `DuplicateFinder` as a pure Core service over an existing `FolderSnapshot`. It groups readable fingerprints by ordinal SHA-256 equality, excludes non-duplicates and unreadable paths, sorts paths and groups deterministically, and returns defensive materialized results.
- `VerificationService` now reuses the same duplicate-grouping authority without changing verification classifications, ordering, unreadable findings, or `HasDifferences` behavior.
- Story 5.2 wired `duplicates <folder>` into the V1 CLI using the existing scanner and reviewed finder, with deterministic duplicate/no-duplicate reporting and established V1 exit mappings.
- The command works for registered or unregistered folders without loading, creating, validating, or mutating catalog state.
- Duplicate detection remains deterministic, catalog-independent, and target-file safe.

### Important Design Decisions

- Keep content grouping in Core and orchestration in CLI: scanner produces the snapshot, `DuplicateFinder` owns membership and ordering, `ReportFormatter` materializes display lines, and CLI owns writers and exits.
- Reuse one duplicate-grouping implementation for on-demand detection and verification compatibility; do not copy grouping logic.
- Qualify groups by raw fingerprint count before distinct ordinal path projection to preserve established verification behavior.
- Sort each path group and the complete group sequence with ordinal comparison; never depend on hash order, culture, filesystem order, or input enumeration.
- Fail closed when scans are incomplete: unreadable paths or scan failures produce no duplicate result.
- Treat catalog independence as no catalog API or persistence participation. A catalog file physically inside the target is scanned as an ordinary file and remains unmodified.
- Materialize formatter output before the first stdout write so scanner, finder, and formatter failures cannot emit partial success output.

### Review Findings That Improved the Implementation

- Story 5.1 review preserved a single materialized current-file set during verification delegation and added focused nested-path ordering and unreadable-only coverage.
- Story 5.2 review extended target-safety proof to directory state, added a true unreadable-only CLI case, covered unexpected scanner and formatter failures, and pinned both root-present and root-disappeared `FileNotFoundException` classification.
- The separate Story 5.2 gate prevented CLI output and error concerns from leaking into the pure Core grouping service.

### Previous Retrospective Follow-Through

Both Epic 4 action items were completed. Story 5.1 pinned deterministic membership, group/path ordering, unreadable exclusion, compatibility, and materialization. Story 5.2 started only after Story 5.1 was done and reviewed, then directly proved catalog independence and preserved the Core/CLI boundary.

### Lessons Learned

- Extracting an established algorithm is safest when compatibility edges are written down before refactoring and verified through the original consumer.
- Determinism is a complete contract: equality comparer, qualification point, distinct projection, inner ordering, outer ordering, and defensive ownership all need explicit tests.
- ?No dependency? and ?read-only? claims need direct evidence: missing and malformed catalog cases, byte sentinels, target file and directory state, and source-boundary inspection.
- A gated stretch story can close cleanly when the committed Core contract is reviewed first and the gate remains a real go/no-go decision.
- Failure-path tests should cover each pipeline stage, not only the most obvious injected service.

### Risks Carried Forward

- Cross-platform catalog paths and relative-separator policy remain deferred; V1 behavior is Windows-focused.
- Unreadable findings still carry paths rather than rich reasons.
- Files can change during a live scan; V1 does not provide realtime monitoring, filesystem snapshots, or transactionally stable directory reads.
- Text already accepted by a failing output writer cannot be rolled back.
- Packaging, clean-environment installation, stakeholder acceptance, and release/deployment evidence are outside story completion and must be assessed by the V1 final readiness check.
